# Production ownership and release runbook

Current-state evidence (2026-07-15): `origin` is `https://github.com/Phozen/booking-system.git`; independent repository inspection reports owner `Phozen`, visibility `PUBLIC`, default branch `main`; `.vercel/project.json` is absent. **Qbook is NOT production-owned until the company transfers and privatizes these assets and attaches the evidence below.**

This runbook separates repository-controlled safeguards from company-admin work. No step below authorizes changing the live GitHub, Vercel, Supabase, Microsoft 365, or DNS environment without an approved change ticket.

## Repository-enforced release contract

- `.github/workflows/ci.yml` runs on every pull request and push to `main` with Node 22, `npm ci`, a high/critical `npm audit` gate, lint, typecheck, unit/security tests, and a production build.
- `.github/workflows/production-release.yml` is manual, rejects non-`main` refs, re-runs all gates, requires approval from the GitHub `production` environment, and promotes an already validated Vercel deployment without rebuilding it.
- `package-lock.json` is authoritative. Release jobs use `npm ci`, never a floating install.
- Production database changes are forward migrations. Migrations `0031_preprovisioned_microsoft_access.sql` and `0032_booking_approval_security.sql` must be applied and verified before the matching application artifact is promoted.

## Release procedure

1. Merge only after `verify` passes and required reviewers approve.
2. Apply required database migrations first using the checklist above; confirm pgTAP and smoke tests.
3. Validate the Vercel preview with Employee, Admin, and Super Admin positive and negative workflows.
4. In GitHub **Actions > Production release > Run workflow**, select `main` and enter the exact validated preview URL.
5. Independent production-environment reviewers compare commit SHA, migration evidence, and preview URL before approving.
6. After promotion, verify `/api/cron/email/run` authentication, `/admin/system-health`, sign-in tenant/allowlist rejection, one approved booking workflow, and logs for new errors.

## Application rollback

List deployments with `vercel ls`, inspect the candidate with `vercel inspect DEPLOYMENT_URL`, and restore the last known-good artifact with `vercel rollback DEPLOYMENT_URL` (or `vercel promote LAST_GOOD_URL`). The Vercel release owner records old/new deployment IDs and post-rollback smoke results. Database rollback is independent: never deploy older application code if it is incompatible with already-applied forward migrations.

## Secret handling

Secrets belong in the company vault and provider environment stores. Evidence must show names, scope, rotation timestamp, and owner - never values. If a secret is found in Git history, immediately revoke/rotate it first, remove it with an approved history-rewrite procedure, invalidate caches/forks where possible, and attach provider revocation evidence.
